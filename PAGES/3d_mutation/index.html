Introduction

The upcoming sections will be devoted to vector mutation. However, to properly cover this subject, we first need to introduce some preliminary concepts. The current section in particular seeks to distinguish between assignments and mutations of variables. For its part, the next section elaborates on vector indexing, providing various flexible methods for selecting elements.

The difference between assignments and mutations can easily go unnoticed by new users: both operations use the operator =, despite being fundamentally different. Moreover, a clear distinction between these concepts is relevant not only for Julia, but also for other programming languages such as Python.

Some Background

Recall that a variable is a label for an object, where the object in turn holds a value. Moreover, objects can have a different number of elements, encompassing both single-element objects (e.g., integers, floating points) and collections (e.g., vectors).

NUMBER

VECTOR

The distinction between an object and its elements is crucial for the rest of the section, as assignments are operations related to objects, whereas mutations are related to elements. Specifically, assignments rebind variables to new objects, while mutations modify the elements of an existing object.

Assignments

Assignments bind a variable to an object. The operation is performed through statements like x = 3 or x = [1,2,3], where 3 and [1,2,3] represent the objects.

Grasping the concept of assignments becomes simpler if we understand an object as a memory address. From this perspective, a statement like x = [1,2,3] can be viewed as a two-step process: i) identifying a memory location that will hold the object with value [1,2,3], and ii) assigning a label x to the memory location for easy access to it.

Warning!
We'll use the analogy of memory addresses to help explain objects. However, memory locations should be understood as an "abstract" storage concept, in the sense that they don't necessarily correspond to a physical location in RAM. Internally, memory is handled in a complex way.

Mutations

As we've indicated, assignments associate the name x to an object. The operation works identically regardless of whether the object is a number or a vector. However, vectors open up the possibility of modifying an object's elements without necessarily creating a new object. [note] Making such a distinction for numbers is unimportant for our purposes. Executing x=2 and then x=3 would ultimately entail a similar use of the operation =. We'll revisit this matter when we cover the more general concepts of mutable and immutable objects in the next section.

This leads us to distinguish between the use of = for assignments (e.g., statements like x = [4,5,6]) and for mutations (e.g., statements like x[1] = 0). The operations are fundamentally different, despite both using the operator =. An assignment changes the object that x is bound to, meaning that it points to a new memory address. On the contrary, a mutation modifies the contents of the object, without changing its memory address.

As an example, suppose that we run x = [4,5]. This creates an object with value [4,5] and stores it in some particular memory location. If we subsequently run x = [6,7], x will be associated with a new object holding [5,6]. This operation constitutes an assignment. Conversely, if we execute x[1] = 0, the operation modifies the object [4,5] and transforms it into [0,5]. The object will still be allocated in the same memory address, and so this case describes a mutation.


MUTATION

ASSIGNMENT


Remark
You can mutate all the elements of x by referring to x[:], without this entailing a new assignment. In other words, modifying the values of x[:] does not change the object that x points to.

Code 

x = [4,5]

x[:] =  [0,0]

Output in REPL
julia> x
2-element Vector{Int64}: 0 0

This becomes important when we study speeding up operations, as usually mutating elements is faster than creating new objects.

Alias vs Copy

A statement like y = x in Julia is interpreted as y pointing to the same object pointed by x. The operation determines that y constitutes an alias of x, so that there are two different labels for the same object. Note that y = x should not be interpreted as y being a label for x. This semantic distinction is subtle, but has deep implications at a practical level. It implies that reassignments of x do not affect y.

To illustrate this, suppose we execute x = 2 and then y = x. This means that both x and y are pointing to the same object holding 2. If we now execute x = 4, x would point to a new object holding the value 4, without changing what's stored in the original object of x. Consequently, y will keep holding the value 2, as y still points to the same object. This is demonstrated below.

x = 2   #'x' points to an object with value 2

julia> y

Remark
Notice that two variables could comprise identical elements, and yet refer to different objects.

This can be formally seen by resorting to the operators == and ===, which assess two types of equality. As for x == y, it evaluates whether x and y have equal values, regardless of whether they're the same object. Instead, x === y indicates whether both x and y are pointing to the same object, interpreted as a specific memory address. By use of these operators, the following example shows that having identical elements does not imply equivalence of objects.

Code 

x = [4,5]

y = [4,5]
z = x

Output in REPL
julia> x == y
true #`x` and `y` have identical elements
julia> x === y
false #`x` and `y` DO NOT point to the same object

julia> x == z
true #`x` and `z` have identical elements
julia> x === z
true #`x` and `z` DO point to the same object

Graphical Representation

We've indicated that the operation y = x creates an alias of x, so that y and x are two different labels for the same object. Consequently, modifying the elements of x or y changes the elements of both x and y simultaneously. This is illustrated in the following diagram and code snippet.

Graphical Representation

y as an Alias 

x = [4,5]
y = x

x[1] =  0

Output in REPL
julia> x
2-element Vector{Int64}: 0 5
julia> y
2-element Vector{Int64}: 0 5

If you want x and y to be considered as separate objects, you should use the function copy instead. This function creates a new object that contains the same elements as the original object.

y as a Copy 

x = [4,5]
y = copy(x)

x[1] =  0

Output in REPL
julia> x
2-element Vector{Int64}: 0 5
julia> y
2-element Vector{Int64}: 4 5
