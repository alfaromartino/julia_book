Introduction

Global variables can be accessed and modified anywhere in the code. In contrast, local variables are only accessible within a specific scope (e.g., in a function or a loop). In particular, we'll exclusively refer to local variables as those in relation to a function's scope, which encompass both function arguments and variables defined inside the function.

Operations with global variables are a common source of type instability: since they can be redefined at any point, Julia treats them as having type Any. This type ambiguity explains why performing operations in the global scope is slow—the compiler has to consider multiple possible methods for each operation, precluding specialization.

This section explores two approaches to making operations with global variables type-stable: type-annotating global variables and declaring global variables as constants. The latter approach is particularly convenient for variables that remain unmodified throughout the script, like π = 3.1415926535897.

Warning! - You Should Always Wrap Code in a Function
Keep in mind that functions implement additional optimizations that aren't possible in the global scope. Due to this, even if you implement the fixes for global variables that we propose in this section, you should additionally rely on functions when you care about performance.

When Are We Using Global Variables?

Self-contained functions solely rely on their arguments and locally defined variables, without depending on global variables. The first tab below shows an example of a self-contained function. In contrast, the subsequent two tabs illustrate scenarios operating with global variables. They include a function employing variables defined outside its scope, as well as operations performed in the global scope. 

x = 2

function foo(x) 
    y = 2 * x 
    z = log(y)

    return z
end

@code_warntype foo(x)  # type stable
x = 2

function foo() 
    y = 2 * x 
    z = log(y)

    return z
end

@code_warntype foo() # type unstable
# all operations are type unstable (they're defined in the global scope)
x = 2

y = 2 * x 
z = log(y)

Achieving Type Stability With Global Variables

Working with global variables is a practice often discouraged in modern programming paradigms, which additionally affects performance by turning operations type unstable. However, there are some scenarios where global variables could come in handy. In that case, there are two approaches addressing the performance issue: setting a global variable as a constant or annotating it with a concrete type.

Note, nonetheless, that writing self-contained functions should be the first option to consider. Aside from performance considerations, these functions maintain the code modular and easier to understand, without the need to track global variables across the script to discern their purpose.

Constant Global Variables

Declaring a global variable as a constant requires adding the keyword const before the variable's name, such as const x = 3. Any variable can be declared as a constant, including vectors, as the following examples show.

const a = 5
foo()   = 2 * a

@code_warntype foo()        # type stable
const b = [1, 2, 3]
foo()   = sum(b)

@code_warntype foo()        # type stable

A constant global variable doesn't incur the performance penalties of using a global variable. However, it's worth remarking that you should declare a global variable as a constant only if its value won't change throughout the script. In other words, the global variable should be truly constant, such as the value \(\pi := 3.1415\). Reassigning the value of a constant at some point in the script will lead to silent bugs.

Warning!
If you have to modify the value of a constant global variable, change the definition of the constant and then re-run the entire script. Redefining the value later in the code will trigger a warning in the REPL. The reason is that functions defined earlier in the code will still rely on the constant's original value. Only if you rerun the whole script will changes take effect

To illustrate it, compare the following code snippets. Both execute the function foo for a constant value of x=1, and then for a value of x=2. The first tab shows the consequences of a script including two different constant values for x, where the value returned by foo is still based on x = 1. On the contrary, rerunning the entire script based on a new value for x circumvents the issue, as demonstrated in the second tab.

const x = 1
foo()   = x
foo()             # it gives 1

x = 2
foo()             # it still gives 1

const x = 2
foo()   = x
foo()             # it gives 2

Redefining constants was introduced to facilitate code testing, as the user doesn't need to start a new Julia session for each new constant value. Despite this, you should be cautious when doing so.

Type-Annotating a Global Variable

The second approach to ensure type stability is by specifying a global variable's type. This is done by including the operator :: after the variable's name, such as x::Int64. Compared to const, this approach enables you to reassign global variable without unexpected consequences, as :: only affects the type of the variable, not its value.

x::Int64 = 5
foo()    = 2 * x

foo()          # type stable

y::Vector{Float64} = [1, 2, 3]
foo()              = sum(y)

foo()          # type stable

Type-annotating a function argument doesn't improve performance
Type-annotating function arguments is commonly done when packages are written. The goal is to ensure that the function accepts only arguments with a specific type. By doing so, the developer shields the user from getting an output despite passing variables with wrong types.

For instance, * applied to strings in Julia will concatenate words. Thus, if we want to define a function to multiple numbers, a way to prevent misuses is by restricting the operation to numeric types.

foo1(a,b) = a * b

a = 1 
b = 2

x = "hello "
y = "world"

Output in REPL
julia> foo1(a,b)
2

julia> foo1(x,y)
"hello world"

foo2(a::Number, b::Number) = a * b

a = 1 
b = 2

x = "hello "
y = "world"

Output in REPL
julia> foo2(a,b)
2

julia> foo2(x,y)
ERROR: MethodError: no method matching foo2(::String, ::String)

Beyond this use, type-annotating function arguments doesn't affect type stability. The reason is that a function always checks the types of its arguments, regardless of whether you type annotate them.

Type-annotating a function argument is only relevant for type stability when keyword arguments employ global variables as their default value. We'll expand on this in a subsequent section.
