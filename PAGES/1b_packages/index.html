Introduction

When you start Julia, only a handful of very basic functions are available (e.g., those to perform sums, products, and subtractions). Common operations, such as plotting or the computation of an average, are not even possible. Julia, instead, relies on what's known as packages to incorporate additional functionalities into the workspace.

Basically, packages can be understood as modules containing a set of functions. Their main goal is to separate the development of the tasks you're focusing on from other functionalities. The approach allows for improvements on specific tasks, without the need of affecting other aspects of Julia's ecosystem. Furthermore, it can be understood as an application of a broader concept in programming, known as modularity.

The practical consequence is that you'll have to load several packages every time you start a new session in Julia. In the following, we start by explaining how to call the functions of a package. After that, we will discuss how to install a package. Although it'd be more logical to start with the installation process, we discuss it later, since this requires the use of a package itself.

Warning!
Handling package syntax may appear confusing at this point, since it relies on Julia's syntax for vectors and functions. As we progress through the website's topics, you'll see that it actually follows the same general logic of the language.

Loading Packages

There are two commands to load packages into your workspace: import and using. When we "import" a package, you have access to the package's functions, but their use requires indicating the package. You can avoid this step, by "using" a package, allowing you to directly call the function.

To illustrate the difference between these concepts, suppose we want to load a standard package in Julia called Statistics. This loads standard statistics functions such as the mean, median, and variance. The example demonstrates how import and using calls the function mean from Statistics to compute the average of some vector x.

Code 

#IMPORT
import Statistics #getting access to the function `mean` will require the prefix `Statistics.`
Statistics.mean(x) 


#USING
using Statistics #no need to add the prefix `Statistics.` to call `mean`(although it's possible to do so)
mean(x)

Why should we use import if it makes the code more verbose? For a start, it becomes necessary when two packages share the same name for a function. And even if this issue doesn't arise, import enables us to read the code more easily, by giving a hint on the function's meaning.

For instance, a function called rank could give rise to some confusion, since it could refer to different concepts (e.g., the rank of a matrix, computing a ranking order). By specifying the package, we reduce the chance of creating an ambiguity.

Remark
import is also useful when you have customized functions and you use them repeatedly across projects. To let the reader know that these functions don't belong to any standard package, you can group them into a package called userFunctions.

For example, one of the functions I have is called table_in_pdf. This translates tables in Julia into some format that is capable of being exported to a pdf. It additionally specifies some settings regarding font, numbers format, etc. While the name of the function makes it clear what it's doing, the user could wonder whether this function is coming from a standard package. You could hint to the user that this isn't the case, by making it available in the workspace through import userFunctions. In this way, you'll export tables to pdf through the function userFunctions.table_in_pdf.

Changing Names and Additional Ways to Load Packages

Regarding using and import, what we've seen is all that you'll probably need to use packages in Julia. However, let me mention that there are additional options that qualify how packages are loaded. For instance, imagine we only need the function mean from Statistics, so that loading the whole package is unnecessary. [note] This is not particularly important for the example used, since loading Statistics is not time-consuming. However, a package could be so heavy that it takes even minutes to fully load. Then, we can do the following.

Code 

import Statistics: mean # you only load `mean`, and use it through `Statistics.mean(x)`
using Statistics: mean  # you only load `mean`, and use it through `mean(x)

We can also change the names used to call either a package or function. This is useful if a name is either too long or used in another function/package.

Code 

#change the name of `Statistics`
import Statistics as st
st.mean(x)            #computes the mean of x

#change the name of `mean`
import Statistics: mean as average
Statistics.average(x) #a way to rename the function `mean

Base Julia

The functions loaded in Julia by default follow the same logic of any package. They're available in a module called Base. Thus, we can think of Base as a package automatically loaded every time we start a session in Julia.

Code 

x = [10, 20, 30] # define an array `x`

#two ways to call `sum`
sum(x)
Base.sum(x)

There are additional functions in Base not loaded by default, but available on request. For instance, it's possible to "import" the function isgreater, which compares two numbers and tells you which one is greater. [note] Why isgreater is not incorporated by default, or its difference relative to >, is unimportant. Just in case you're curious, isgreater can handle comparisons involving a data type handling missing values, while > can't.

Code 

isgreater(2,1) #this errors because the function is not available

import Base: isgreater
isgreater(2,1) #this works and gives `true` as resul
