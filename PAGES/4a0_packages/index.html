Introduction

Functions can be broken down into three:

  1. built-in functions loaded by default,

  2. third-party functions, and

  3. user-defined functions.

This section focuses on i) and ii), relegating iii) to the next section. We consider in particular how to apply functions, which in turn takes us to the discussion of packages.

Packages

When you start a new session in Julia, only a handful of very basic functions are available (e.g., those for sums, products, and subtractions). This is a deliberate choice made by Julia developers, who rely on packages for incorporating additional functions into the workspace. Both built-in functions and third-party developer functions are contained in packagesâ€”the only difference is that built-in functions are loaded by default.

The approach is not unique to Julia. However, Julia embraces this philosophy more than other programming languages. Thus, it doesn't even include standard functions such as averages or standard deviations, which are instead relegated to a package called Statistics. [note] The extent to which Julia advocates for this principle is evident in Statistics itself, where functions for computing distributions are included in another package, called Distributions.

The logic is based on a programming principle known as modularity. This promotes the development of small reusable modules, rather than large intertwined code. Its main advantage is letting packages evolve independently, without bugs and deprecations spreading across the entire Julia ecosystem. The practical implication for users is that it becomes necessary to load several packages in each session, even to perform simple tasks.

Loading Packages and Calling Functions

The concept of packages is tightly related to modules, as they are a particular case of the latter. Formally, modules are independent blocks of code, each acting as a separate workspace and exporting a set of functions. When you run Julia, you're in fact writing your script in a module called Main.

Packages are a special types of modules that hold information about its dependencies, which refer to the necessary packages to run the package itself. To get access to a package's functions, you need to load by using either the keyword import or using. The choice between them determines how the functions are eventually called: if you load a package with import, you need to prefix the function's name with the package's name; instead, no prefix is needed when the package is loaded with use.

Below, we demonstrate the syntax of each approach. For the example, we call the function mean from the package Statistics. [note] You don't need to install the package Statistics. This comes pre-installed with Julia, although it isn't loaded by default. 

x = [1,2,3]

import Statistics   #getting access to its functions will require the prefix `Statistics.`
Statistics.mean(x)

x = [1,2,3]

using Statistics    #no need to add the prefix `Statistics.` to call its functions (although it's possible to do so)
mean(x)

Built-in Functions

Julia's built-in functions come from several packages loaded by default, known as Core and Base. These modules are loaded automatically in every Julia session. In particular, they are incorporated to the workspace as if we had executed using Core and using Base. This determines that Julia's built-in functions can be called without requiring any prefix. [note] Strictly speaking, some built-in functions could require a prefix. For instance, this occurs with the function isgreater, which has to be called through Base.isgreater. Furthermore, there are sub-modules loaded by default acting as subpackages. For instance, you have the function Base.Iterators.accumulate, which is part of the sub-module Iterators from Base. As Base is loaded by using, you can directly call this function through Iterators.accumulate.

Why Using "Import" if It's More Verbose?

When a function's name is shared across two or more packages, you necessarily have to load the packages using import. For instance, suppose we load Statistics and another package called MyPackage. Assume also that the latter contains a function called mean. Then, we need to use import to load either Statistics or MyPackage if we want to use mean. Otherwise, Julia will throw an error when we call mean. [note] Adding a function with the same name as another package isn't necessarily a neglect, but actually a deliberate choice. For example, mean in MyPackage could be computing the average utilizing an alternative algorithm to the one in Statistics.

Loading packages with import could also be beneficial in other scenarios. For instance, it may reduce the possibility of ambiguity in the meaning of a function, thereby improving code readability. For instance, consider a script making use of the function rank. This name could be a reference to various concepts depending on the context (e.g., the rank of a matrix, the order in a list). In this context, you can shed some light on its particular meaning by indicating the package that rank belongs to when you call the function.

Remark
import is also useful if you have customized functions that are commonly applied across projects. For example, I have a function called table_in_pdf that I regularly use in my projects. The function exports Julia tables to a pdf with some predefined format. While the name of the function makes it clear what it's doing, an user could wonder if this function comes from a standard package. You could hint this isn't the case, by including the function in a package called userDefined. In this way, you could load the package through import UserDefined, and then calling the function through UserDefined.table_in_pdf.

Names of Functions and Approaches for Loading Packages (OPTIONAL)

The features we've covered so far are all you probably need to use packages in Julia. However, there are some additional features worth mentioning.

First, it's possible to only load a subset of functions from a package. This feature is particularly relevant for heavy packages that may take some time to fully load. For instance, suppose we only require the function mean from the package Statistics. The following two approaches achieve this and are equivalent.

x = [1,2,3]

import Statistics: mean 
mean(x)                   # no prefix needed

x = [1,2,3]

using Statistics: mean
mean(x)

Note that this approach deems unnecessary adding the package's name as a prefix, even when we load a package through import.

Related to this is the ability to assign custom names to packages or functions. This is particularly useful when the original name of the package or function is lengthy. We demonstrate its implementation through the keyword import, noting that using could be used alternatively. 

x = [1,2,3]

import Statistics as st
st.mean(x)

x = [1,2,3]

import Statistics: mean as average
average(x)                   # no prefix needed

using Statistics: mean as average
average(x)

Notice again that the function's name with import doesn't require any prefix when it's called.
